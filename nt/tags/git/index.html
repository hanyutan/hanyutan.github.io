    Introduction

    如果你不会Web前端开发，也不愿意花钱买服务器，懒洋洋地做个伸手党，只希望专注于内容创作…… 那么Github Pages + Hugo绝对适合你！

    Why Github Pages?

    免费！创建和管理都很方便！

    只需在Github上创建一个repository，然后将Hugo生成的静态页面相关文件push到项目上。 （个人主页<username>.github.io的话push到默认的master分支上，普通repository则push到该项目的gh-pages分支上并保证根目录有一个index.html文件）

    Push的方法有多种，说一些我接触到的：

    • 使用Git命令（最原始，最基本，灵活度高）
    • 下载GitHub桌面版，2次按键搞定（最简单，灵活度低）
    • 基于Git使用sublime上传
    • Eclipse里自带Git，但需要Java环境，jre或jdk都行。

    Why Hugo?

    目前流行的几个静态网站生成器（SSG——Static Site Generator）：Hugo/Hexo/Jekyll。

    Hugo官方主页：https://gohugo.io

    • Hugo是使用Go语言编写的静态网站生成器。
    • 相对于Jekyll复杂的安装设置来说，Hugo进行需要下载一个二进制文件（hogo.exe）即可轻松调试和生成静态页面。
    • Hugo生成静态页码的效率很高，速度上来说Hugo>Hexo>Jekyll。
    • Hugo自带watch的调试模式，可以在修改Markdown的文章内容的同时，自动更新浏览器页面呈现最终效果。
    • 缺点是主题模板相对其他两个来说数量比较少。
    • 用的人感觉还不算特别多，中文社区支持也不是很多。

    Commit History

    git add test.txt
git commit -m "append GPL"

# 查看历史纪录
git log --pretty=oneline --abbrev-commit
#18a6b3f (HEAD -> master) append GPL
#8742dec add distributed
#c0a9a57 wrote a test file

    再次修改本地文件test.txt内容为：
    Git is a distributed version control system.
    Git is a free software distributed under the GPL.

    提交修改后，使用log命令查看历史纪录（按Q结束）。如果近运行命令git log将返回详细内容：
    commit 18a6b3fd4235fab6bc0e205c921e5b43a1695339 (HEAD -> master)
    Author: user.name user.email
    Date: Wed Jul 31 16:37:58 2019 +0800
              append GPL

    其中第1行的一大串数字的字母叫做commid id（版本号），HEAD可以理解为指针，当前正指向master分支； 第2行的内容是之前下载Git后使用git config --global指定的用户名与Email； 第3行则是提交commit的时间；最后一行为提交的commit内容。

    使用git log --pretty=oneline可返回缩短到一行的简洁版，
    使用git log --pretty=oneline --abbrev-commit可返回commit id也缩短的简洁版。

    Rollback / Reset

    # 回退到上一版本
git reset --hard HEAD^
#HEAD is now at 8742dec add distributed

# 查看文件内容
cat test.txt

# 回到指定版本
git reset --hard 18a6b3f

    HEAD是当前版本，HEAD^^代表上上一个版本，类似的HEAD~100是往上100个版本。 此时使用cat命令查看文件内容，会发现是没有进行最后一次修改的内容。
    Git is a distributed version control system.
    Git is a free software.
    add something

    此外还可以使用reset命令回到指定版本，或者取消回退而继续最新版本。 注意commit id可以在log中找到，并且没必要写全，只写前几位即可。 另外，假设命令窗口不小心被关掉了，git log不再起作用，可以使用git reflog追踪commit id。

    Working Directory & Stage

    工作区working directory是在电脑里能看到的目录，比如Blog_Git文件夹就是一个工作区。 工作区有一个隐藏目录.git是Git的版本库repository，里面存了很多东西，其中最重要的就是暂存区stage/index， 还有Git自动创建的第一个分支master，以及指向master的指针HEAD。

    git add把文件修改添加到暂存区；git commit把暂存区的所有内容提交到当前分支。 简单理解为：需要提交的文件修改通通放到暂存区，然后一次性提交暂存区的所有修改。

    # 查看工作区和版本库里面最新版本的区别
git diff HEAD -- test.txt

    Git跟踪并管理的是修改而非文件，举例：第一次修改 -> git add -> 第二次修改 -> git commit。 当你用git add命令后，在工作区的第一次修改被放入暂存区，准备提交但是， 在工作区的第二次修改并没有放入暂存区，所以git commit只负责把暂存区的修改提交了， 也就是第一次的修改被提交了，第二次的修改不会被提交。

    Undo

    # 撤销工作区的修改
 git checkout -- test.txt
# 撤销add操作，重新放回工作区
git reset HEAD test.txt

    使用checkout命令把test.txt文件在工作区的修改全部撤销，这里有两种情况：
    一种是test.txt自修改后还没有被放到暂存区，现在撤销修改就回到和版本库一模一样的状态；
    一种是test.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。
    总之，就是让这个文件回到最近一次git add或git commit时的状态。
    注意命令中如果没有--就变成了“切换到另一个分支”的命令

    总结一下有几种情况：
    1. 刚在文件中做了修改，如何撤销？
    git checkout – test.txt
    2. 修改的文件已经add，如何撤销？
    git reset HEAD test.txt
    git checkout – test.txt
    3. 修改的文件已经add、commit，但未推送到远程库，如何撤销？
    git reset –hard HEAD^
    4. 修改的文件add后又做了修改，如何撤销？
    git checkout – test.txt

    Delete Files

    rm test.txt                            # 删除文件
# git checkout -- test.txt             # 文件误删需要恢复
git status                             # 工作区和版本库不再一致
git rm test.txt                        # 确定要从版本库中删除文件
git commit –m “remove test.txt”        # 提交删除操作

    注意git checkout其实是用版本库里的版本替换工作区的版本，无论工作区是修改还是删除， 都可以“一键还原”，但从来没有被添加到版本库就被删除的文件，是无法恢复的。

    Remote Repo & SSH Key

    Git是分布式版本控制系统，同一个Git仓库，可以分布到不同的机器上（多个克隆的版本库）。 一台电脑充当服务器的角色，每天24小时开机，其他每个人都从这个“服务器”仓库克隆一份到自己的电脑上， 并且各自把各自的提交推送到服务器仓库里，也从服务器仓库中拉取别人的提交。 只要注册一个GitHub账号，就可以免费获得Git远程仓库。 由于你的本地Git仓库和GitHub仓库之间的传输是通过SSH加密的，所以，需要一点设置SSH Key。

    第1步：打开Git Bash创建SSH Key。ssh-keygen -t rsa -C "youremail@example.com"
    替换自己的邮件地址，然后一路回车，无需设置密码。随后显示：
    Your identification has been saved in /d/Github/.ssh/id_rsa.
    Your public key has been saved in /d/Github/.ssh/id_rsa.pub.
    ……

    第2步：登陆GitHub右上角打开Settings，SSH and GPG keys条目，点击“New SSH Key”， 填上任意Title，在Key文本框里粘贴id_rsa.pub文件的内容，点击Add SSH key。

    为什么GitHub需要SSH Key呢？因为GitHub需要识别出你推送的提交确实是你推送的，而不是别人冒充的。 Git支持SSH协议，所以GitHub只要知道了你的公钥，就可以确认只有你自己才能推送。当然，GitHub允许你添加多个Key。 假定你有若干电脑，你一会儿在公司提交，一会儿在家里提交，只要把每台电脑的Key都添加到GitHub，就可以在每台电脑上往GitHub推送了。 GitHub上免费托管的Git仓库，任何人都可以看到，但只有你自己才能改。如果你不想让别人看到Git库，有两个办法， 一个是交点保护费让GitHub把公开的仓库变成私有的（别人看不见，不可读更不可写）； 另一个办法是自己动手，搭一个自己的Git服务器，别人也是看不见的，公司内部开发必备。

    Add Remote Repo

    git remote add origin https://github.com/user/repo.git
git push -u origin master                  把本地库的内容推送到远程

    已经在本地创建了一个Git仓库后，想要关联到GitHub的远程仓库进行同步，之后在本地修改后推送到远程。

    用git remote命令添加远程库，默认名字是origin（可以修改）； 用git push命令，把当前分支master推送到远程。 注意本地库可以同时与多个远程库互相同步，但需要保证远程库的名称不同。

    由于远程库是空的，我们第一次推送master分支时需要加上-u参数， Git不但会把本地的master分支内容推送的远程新的master分支， 还会把本地的master分支和远程的master分支关联起来， 在以后的推送或者拉取时就可以简化命令了。

    推送成功后，可以立刻在GitHub页面中看到远程库的内容已经和本地一模一样。 从现在起，只要本地作了提交，就可以通过命令：git push origin master把本地master分支的最新修改推送至GitHub。 分布式版本系统的最大好处之一是在本地工作完全不需要考虑远程库的存在，当有网络的时候，再把本地提交推送一下就完成了同步。

    如果遇到报错fatal: The remote end hung up unexpectedly

    可以在版本库的隐藏文件夹.git中修改config文件，添加以下内容
    [http]
    postBuffer = 524288000
    或者直接执行命令：
    git config http.postBuffer 524288000 修改单个仓库配置到500M
    git config –global http.postBuffer 1048576000 修改全局配置到1G
    其中524288000–>1024*1024*500

    如果出现如下错误：curl 56 OpenSSL SSL_read:SSL_ERROR_sysCALL

    执行：git config http.sslVerify «false»

    Branch

    创建与合并分支

    git checkout -b dev                  # 创建并切换到dev分支，命令行末尾显示dev

git branch -d dev                    # 合并完成后删除dev分支
git branch                           # 只剩下master一个分支

    创建属于自己的分支，其他成员在原来的分支上工作， 而你在自己的分支上干活不会影响到其他人，开发完毕后再一次性地合并到原来的分支上。

    (1)每次提交文件及修改，Git都把它们串成一条时间线，这条时间线就是一个分支。 master分支是主分支，每次提交，master分支都会向前移动一步，Git用master指向最新的提交， 再用HEAD指向当前分支master，由此master分支的线越来越长。
    (2)创建新分支时（例如dev），Git新建一个指针dev，指向master相同的提交，再把HEAD指向dev，代表当前分支dev。
    (3)从现在开始，对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev指针往前移动一步，而master指针不变。
    (4)假如我们在dev上的工作完成了，就可以把dev合并到master上。 注意到合并后返回的信息，Fast-forward告诉我们这次合并是“快进模式”，也就是直接把master指向dev的当前提交，所以合并速度非常快。
    (5)合并完分支后，甚至可以删除dev分支，把dev指针给删掉，这样就剩下一条master分支。
    (6)切换回master分支后，查看test2.txt文件发现刚才添加的内容不见了。因为那个提交是在dev分支上，而master分支此刻的提交点并没有变。

    因为创建、合并和删除分支非常快，所以Git鼓励你使用分支完成某个任务，合并后再删掉分支，这和直接在master分支上工作效果是一样的，但过程更安全。

    git checkout –b dev相当于以下两条命令: git branch dev 和 git checkout dev
